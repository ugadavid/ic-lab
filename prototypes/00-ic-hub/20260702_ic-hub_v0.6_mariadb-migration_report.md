# IC-Lab Hub V0.6 - Premiere couche MariaDB

Date : 2026-07-02

## Objectif

La V0.6 prepare le hub IC-Lab a sortir durablement du stockage JSON local en ajoutant une premiere couche MariaDB, sans casser le prototype V0.5. Le mode JSON reste disponible et reste le mode par defaut.

## Pourquoi MariaDB plutot que SQLite

MariaDB est deja present dans l'environnement de travail pour Dico-IC. Le choix evite d'introduire une nouvelle technologie de stockage et prepare une architecture plus proche d'un futur hub multi-utilisateurs. SQLite aurait ete plus simple localement, mais moins aligné avec l'infrastructure deja en place.

## Base utilisee

- Hote : `127.0.0.1`
- Port : `3306`
- Base : `ic_hub`
- Utilisateur : `ic_hub_user`

La base `ic_dico` et les tables Dico-IC ne sont pas modifiees.

## Donnees migrees

Le script `node db/migrate-json-to-mariadb.js` lit les stores JSON existants et fait des upserts idempotents avec `INSERT ... ON DUPLICATE KEY UPDATE`.

- `run_events`: 16 apres test fonctionnel MariaDB

## Mode JSON / mode MariaDB

Mode JSON par defaut :

```powershell
node server.js
```

Mode MariaDB :

```powershell
$env:IC_HUB_STORE="mariadb"
node server.js
```

Le serveur lit `IC_HUB_STORE`. Si MariaDB est indisponible au demarrage ou pendant une operation, le serveur log l'erreur et retombe temporairement en JSON.

## Securite et prudence

- Les mots de passe ne sont pas migres en clair : `passwordHash` et `passwordSalt` sont conserves.
- Les sessions stockent `token_hash` en MariaDB.
- Les runs stockent seulement `launch_token_hash`.
- Les evenements de run gardent les payloads filtres, sans texte apprenant complet.
- Les fichiers JSON ne sont pas supprimes.
- Aucun `DROP DATABASE` ni `DROP TABLE` automatique.

## Limites

- Le mode MariaDB est experimental en V0.6.
- La couche store conserve les formes JSON internes pour limiter les regressions.
- Les dates sont stockees comme chaines ISO pour cette premiere passe.
- Les migrations n'effacent pas les donnees absentes des JSON afin d'eviter les pertes accidentelles.
- Les transactions applicatives fines restent a ajouter.
- Les pages V0.6 sont essentiellement les pages V0.5 avec backend compatible MariaDB.

## Prochaines pistes

- Ajouter des transactions sur les operations run/event et assignation/ownership.
- Remplacer progressivement les stores JSON internes par des repositories SQL plus explicites.
- Ajouter une commande de verification de coherence post-migration.
- Ajouter une strategie de purge/archivage des sessions.
- Deplacer les secrets vers une gestion d'environnement plus robuste.
