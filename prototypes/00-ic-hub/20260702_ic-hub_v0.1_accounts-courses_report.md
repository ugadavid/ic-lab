# IC-Lab Hub V0.1 - comptes, roles, cours

## Objectif

Cette V0.1 prouve qu'IC-Lab peut disposer d'un hub central independant des prototypes. Le hub organise des comptes, des roles, des cours, des inscriptions et des assignations d'activites sans enfermer cette logique dans le Prototype 06.

## Pourquoi separer le hub du Prototype 06

Le Prototype 06 reste un module branchable : il conserve sa bibliotheque, son compositeur et son lecteur. Le hub vit au-dessus des prototypes et pourra plus tard router vers Seven Sieves, Dico-IC, Video augmentee IC et Rencontre plurilingue REPLI4C.

## Architecture serveur

Le serveur Node natif se trouve dans `prototypes/00-ic-hub/server/`. Il sert les pages de `public/` et expose une API REST sous `/api` sur le port `8790`.

Le stockage est local et JSON :

- `users.json`
- `sessions.json`
- `courses.json`
- `enrollments.json`
- `course-activities.json`
- `prototypes.json`

## Roles

Roles prevus :

- `admin`
- `teacher`
- `student`
- `observer`

La V0.1 implemente surtout `admin`, `teacher` et `student`. `observer` existe comme role demo mais reste peu exploite.

## Comptes demo

Au premier lancement, si aucun utilisateur n'existe, le serveur cree :

- `admin@demo.local` / `admin`
- `teacher@demo.local` / `teacher`
- `student@demo.local` / `student`
- `observer@demo.local` / `observer`

Les mots de passe ne sont pas stockes en clair. Le serveur utilise `crypto.scryptSync` avec un sel par utilisateur.

## Authentification

Endpoints :

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

Le login retourne un token stocke cote frontend dans `localStorage` sous `icHubAuthToken`. Les routes protegees attendent `Authorization: Bearer <token>`.

## Cours

Endpoints :

- `GET /api/courses`
- `POST /api/courses`
- `GET /api/courses/:id`
- `PUT /api/courses/:id`
- `DELETE /api/courses/:id`

Regles simples :

- admin voit tous les cours ;
- teacher voit et gere ses cours ;
- student voit les cours auxquels il est inscrit.

## Inscriptions et assignations

`POST /api/courses/join` permet a un etudiant de rejoindre un cours avec un code d'acces.

Les activites de cours utilisent :

- `GET /api/courses/:courseId/activities`
- `POST /api/courses/:courseId/activities`
- `DELETE /api/courses/:courseId/activities/:assignmentId`

Une assignation stocke seulement `prototypeId`, `activityId`, `activitySource`, `courseId` et `assignedBy`. Le hub ne copie pas encore la logique interne du Prototype 06.

## Prototypes disponibles

`GET /api/prototypes` expose :

- Prototype 06 - Rencontre plurilingue REPLI4C ;
- Seven Sieves ;
- Dico-IC ;
- Video augmentee IC.

Le Prototype 06 pointe vers `../../06-voice-agent-ic/library-1.1.html` comme module branchable.

## Pages creees

- `login.html` : connexion et comptes demo.
- `hub.html` : accueil connecte, prototypes et cours.
- `teacher.html` : creation de cours et assignation simple d'activites.
- `student.html` : inscription par code et lancement d'activites.
- `admin.html` : utilisateurs, cours, prototypes, sessions et etat serveur.

## Limites de securite

Cette authentification est une demonstration locale. Elle n'est pas prete pour un deploiement public.

Limites connues :

- pas de HTTPS ;
- pas de cookies securises ;
- pas de CSRF ;
- gestion simple des tokens ;
- pas de recuperation de mot de passe ;
- comptes demo faibles ;
- stockage JSON local ;
- permissions simplifiees.

## Prochaines pistes

Les prochaines versions pourront ajouter une vraie gestion de comptes, des roles plus fins, des cours persistants plus robustes, une selection directe des activites du Prototype 06, puis des connecteurs vers Seven Sieves, Dico-IC et Video augmentee IC.
