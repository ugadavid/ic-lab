# IC-Lab Hub V0.6.1 - Durcissement MariaDB

Date : 2026-07-02

## Objectif

La V0.6.1 consolide le mode MariaDB ajoute en V0.6 sans ajouter de fonctionnalite visible majeure. Le hub garde le mode JSON par defaut et le mode MariaDB via `IC_HUB_STORE=mariadb`.

## Transactions ajoutees

Des chemins transactionnels MariaDB ont ete ajoutes pour les operations sensibles :

- `POST /api/runs/start` : insertion atomique du run.
- `POST /api/runs/:runId/events` : verrou SQL sur le run, insertion de l'evenement, mise a jour du statut, de `startedAt`, `completedAt` et `durationMs` dans la meme transaction.
- assignation d'activite a un cours : creation de `course_activities` et creation ou reutilisation de `activity_ownership` dans une transaction.

Le mode JSON conserve son fonctionnement existant avec les verrous d'ecriture fichier.

## Script de coherence

Fichier cree :

- `server/db/check-db-consistency.js`

Commande :

```powershell
node db/check-db-consistency.js
```

Le script verifie les counts principaux, les runs sans evenements, les evenements sans run, les assignations orphelines, les inscriptions orphelines, les ownership sans prototype connu et les doublons potentiels `prototypeId + activityId + activitySource`.

## Endpoint health DB

`GET /api/health` et `GET /api/health/db` retournent maintenant :

- store actif ;
- store demande ;
- version du hub ;
- en mode MariaDB : base, host, port, counts rapides et dernier message d'erreur DB si present.

Aucun mot de passe n'est expose.

## Limites restantes

- Les tables ne declarent pas encore de cles etrangeres SQL strictes.
- Les operations non critiques restent sur l'abstraction store compatible JSON.
- Le mode MariaDB reste une couche de transition avant des repositories SQL plus explicites.
- Les dates restent stockees sous forme de chaines ISO.
