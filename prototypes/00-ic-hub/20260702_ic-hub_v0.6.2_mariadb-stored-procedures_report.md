# IC-Lab Hub V0.6.2 - Procedures stockees MariaDB

Date : 2026-07-02

## Objectif

La V0.6.2 deplace une partie des invariants critiques du hub vers MariaDB. Le but n'est pas de tout mettre en procedures stockees, mais de faire porter a la base `ic_hub` les mutations ou la coherence minimale est importante.

Le mode JSON reste disponible et le mode MariaDB reste active par `IC_HUB_STORE=mariadb`.

## Pourquoi ajouter des procedures stockees

La V0.6.1 avait deja ajoute des transactions cote Node. Cette passe renforce le mode MariaDB en placant aussi dans la base :

- l'atomicite des creations de run ;
- la coherence `runs` + `run_events` ;
- la coherence `course_activities` + `activity_ownership`.

Cela limite le risque qu'un futur client ou script ecrive ces tables critiques en oubliant une partie de la logique metier minimale.

## Procedures creees

Fichier cree :

- `server/db/procedures.sql`

Routines installees :

- `sp_create_run` : valide les champs essentiels et insere un run avec statut `created`.
- `sp_append_run_event` : verifie le run, insere l'evenement, met a jour le statut, `started_at`, `completed_at` et `duration_ms`.
- `sp_assign_activity_with_ownership` : insere l'assignation, cree ou reutilise le record ownership correspondant a `prototype_id + activity_id + activity_source`.
- `vw_activity_ownership_full` : vue diagnostique ownership/prototype/owner/institution.

Les procedures utilisent `START TRANSACTION`, `COMMIT`, `ROLLBACK` et `RESIGNAL` en cas d'erreur.

## Appels cote Node

En mode MariaDB, `server/stores/mariadbStore.js` appelle maintenant :

- `CALL sp_create_run(...)` depuis la creation de run ;
- `CALL sp_append_run_event(...)` depuis l'ajout d'evenement ;
- `CALL sp_assign_activity_with_ownership(...)` depuis l'assignation activite + ownership.

Le helper transactionnel Node ajoute en V0.6.1 n'est plus utilise pour ces trois operations.

## Ce qui reste cote Node

Volontairement, le serveur Node garde :

- authentification ;
- validation de role ;
- verification d'acces au cours ;
- hash du `launchToken` ;
- verification du `launchToken` avant appel procedure ;
- filtrage des payloads apprenants ;
- logique de visibilite effective ;
- compatibilite et fallback JSON.

## Migration et verification

`server/db/migrate-json-to-mariadb.js` execute maintenant :

1. `schema.sql`;
2. `procedures.sql`;
3. les upserts JSON vers MariaDB.

Script cree :

- `server/db/check-db-routines.js`

Il verifie `sp_create_run`, `sp_append_run_event`, `sp_assign_activity_with_ownership` et la vue diagnostique via `INFORMATION_SCHEMA`.

## Limites restantes

- Les tables n'ont pas encore de cles etrangeres SQL strictes.
- La verification du `launchToken` reste cote Node pour eviter de transmettre le token brut a la procedure.
- La visibilite effective reste cote Node.
- Les procedures couvrent les operations critiques prioritaires, pas tout le store MariaDB.
- Les dates restent stockees sous forme de chaines ISO pour compatibilite prototype.
