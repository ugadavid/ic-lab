# Prototype 06 V1.1.2 - Evenements de run hub

Date : 2026-07-02

## Objectif

Permettre a Prototype 06 de participer a une premiere trace de session lancee depuis le hub IC-Lab, sans ajouter de backend, sans IA, et en conservant la logique V1.1.1 de lancement par URL.

## Fichiers crees

- `index-1.1.2.html`
- `script-1.1.2.js`
- `style-1.1.2.css`
- `20260702_proto06_v1.1.2_hub-run-events_report.md`

## Parametres URL pris en charge

- `activityId`
- `activitySource`
- `courseId`
- `assignmentId`
- `runId`
- `launchToken`

## Evenements envoyes au hub

Endpoint :

`POST http://127.0.0.1:8790/api/runs/:runId/events`

Evenements :

- `proto_loaded`
- `activity_loaded`
- `meeting_started`
- `user_answer_submitted`
- `activity_completed`
- `error`

Pour `user_answer_submitted`, Prototype 06 transmet uniquement `answerLength` et `hasText`. Le texte complet de l'apprenant n'est pas envoye.

## Comportement en erreur

- Si `runId` ou `launchToken` manque, l'envoi est un no-op.
- Si le hub est indisponible, l'erreur est attrapee et la rencontre continue.
- En mode `proto06Debug`, le panneau debug affiche le `runId`, la presence du token, les derniers evenements emis et les erreurs d'envoi.
- La valeur du `launchToken` n'est pas affichee.

## Verifications effectuees

- `node --check script-1.1.2.js`
- `GET http://127.0.0.1:8788/index-1.1.2.html?activityId=climateObservations001&activitySource=server` retourne `200`.
- Verification que `index-1.1.2.html` charge `script-1.1.2.js`.
- Verification cote hub que l'URL lancee contient `runId` et `launchToken`.
- Verification cote hub que les evenements attendus sont acceptes et traces.
- Verification cote hub que le mauvais token retourne `401`.

## Limites restantes

- Les envois sont faits en `fetch` simple, sans file d'attente persistante cote navigateur.
- Le token de lancement est present dans l'URL pour cette version locale.
- Les traces restent des evenements pedagogiques minimaux, pas une analytique fine.
