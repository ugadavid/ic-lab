# IC-Lab Hub V0.1.1 - stabilisation serveur

## Cause du crash

Le crash observe (`ERR_HTTP_HEADERS_SENT`) venait d'un serveur trop fragile face aux erreurs asynchrones : le catch global essayait de renvoyer une reponse JSON meme si une route avait deja commence a repondre. En parallele, la lecture du body JSON pouvait rester ouverte sur certains corps invalides, ce qui provoquait cote frontend un `Failed to fetch` ou un timeout.

## Fichiers modifies

- `server/server.js`
- `server/package.json`
- `server/README.md`

## Protections ajoutees

- `sendJson()` verifie `headersSent` et `writableEnded` avant d'ecrire.
- Le catch global renvoie une erreur JSON seulement si la reponse est encore disponible.
- Les erreurs `JSON invalide` et `Payload trop volumineux` retournent `400`.
- `readBody()` valide explicitement le JSON et possede un garde-fou contre les corps non termines.
- Les fichiers JSON attendus sont verifies et crees au demarrage si absents.
- Les lectures/ecritures JSON journalisent les erreurs et retournent des messages lisibles.
- `verifyPassword()` evite les erreurs de longueur dans `timingSafeEqual`.
- Logs minimaux au demarrage, sur login reussi/echoue et sur erreur serveur.

## Resultats de verification

- `node --check server.js` : OK.
- Login admin : OK.
- Login teacher : OK.
- Login student : OK.
- Mauvais mot de passe : `401` propre, sans crash.
- Token invalide : `401` propre, sans crash.
- JSON invalide : `400` propre, sans crash.
- Serveur actif apres erreur : OK.

## Limites restantes

Cette stabilisation ne transforme pas le hub en plateforme de production. Les limites V0.1 restent valables : pas de HTTPS, pas de cookies securises, pas de CSRF, tokens simples, stockage JSON local, comptes demo faibles et permissions simplifiees.
