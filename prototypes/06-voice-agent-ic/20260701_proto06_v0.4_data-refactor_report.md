# Prototype 06 — Rapport V0.4

## Objectif de l'itération

La V0.4 ne cherche pas à modifier l'interface ou l'expérience principale du prototype.

Son objectif est de rendre le prototype plus extensible en séparant les données de la logique d'affichage :

- langues ;
- lieux ;
- personnages ;
- scénarios.

Cette séparation prépare le prototype à accueillir davantage de rencontres plurilingues sans devoir réécrire le script principal à chaque ajout.

## Point de départ

La V0.4 repart de la V0.3.5 incarnée.

Cette base contient déjà :

- personnages visibles ;
- fonds par pays ;
- scène collective finale ;
- effets subtils de présence ;
- contrôles audio ;
- synthèse vocale ;
- reconnaissance vocale.

La V0.4 conserve cette expérience et travaille surtout l'organisation interne des données.

## Problème traité

Dans les versions précédentes, Clara, Marco et Ana étaient encore fortement câblés dans le scénario et dans le script.

La V0.4 prépare l'ajout futur de :

- nouveaux personnages ;
- nouvelles langues romanes ;
- français ;
- roumain ;
- catalan ;
- occitan ;
- portugais européen ;
- langues romanes minoritaires ;
- plusieurs personnages par langue.

## Objets de données créés

### `languageCatalog`

Catalogue des langues disponibles pour les rencontres.

Il centralise les informations utiles à la synthèse vocale et à l'identification pédagogique de la langue.

Exemple :

```js
es: {
  id: "es",
  label: "Espagnol",
  speechLang: "es-ES",
  family: "romane",
  status: "majoritaire",
  notes: "Langue romane utilisée dans REPLI4C."
}
```

### `placeCatalog`

Catalogue des lieux associés aux témoignages.

Il permet de relier un personnage à un contexte visuel et climatique.

Exemple :

```js
valencia: {
  id: "valencia",
  label: "Valencia",
  country: "Espagne",
  background: "images/Espagne.png",
  climateTheme: "étés plus longs et plus chauds",
  notes: "Lieu associé au témoignage de Clara."
}
```

### `characterCatalog`

Catalogue des personnages.

Il relie chaque personnage à une langue, un lieu, une image, un rôle et quelques tags pédagogiques.

Exemple :

```js
clara: {
  id: "clara",
  displayName: "Clara",
  languageId: "es",
  placeId: "valencia",
  image: "images/Clara.png",
  role: "Étudiante",
  shortBio: "Étudiante à Valencia.",
  voiceProfile: {
    preferredLang: "es-ES"
  },
  tags: ["chaleur", "été", "Espagne"]
}
```

### `scenarioCatalog`

Catalogue des scénarios disponibles.

Un scénario référence les personnages par `id` et contient les étapes de prise de parole, la question commune, les observations IC et les mots-clés d'analyse.

Exemple :

```js
participants: ["clara", "marco", "ana"]
```

Chaque étape référence ensuite un personnage :

```js
{
  characterId: "clara",
  text: "En Valencia los veranos son más largos y más calurosos.",
  transparentWords: "veranos, largos, calurosos",
  next: "Ensuite : Marco témoignera en italien."
}
```

### `activeScenarioId`

Identifiant du scénario actuellement chargé par le prototype.

Exemple :

```js
const activeScenarioId = "repli4cClimateRoundtable";
```

## Fonctions JS adaptées

### `getCatalogItem()`

Récupère une entrée dans un catalogue et signale clairement si l'identifiant demandé n'existe pas.

### `buildParticipantStep()`

Reconstruit une étape de rencontre à partir :

- du scénario ;
- du personnage ;
- de la langue ;
- du lieu.

Cette fonction évite de dupliquer dans le scénario le nom, la langue, l'image ou le fond.

### `buildScenario()`

Construit le scénario actif à partir de `scenarioCatalog` et de `activeScenarioId`.

Elle transforme les étapes déclaratives en participants prêts à être affichés.

### `setActiveVisual()`

Met à jour la scène selon le participant actif.

La fonction récupère désormais l'image du personnage et le fond depuis les données reconstruites, plutôt que depuis des chemins codés en dur dans la logique.

### `renderCollectiveCharacters()`

Alimente la scène collective finale à partir des participants reconstruits.

Les trois images de groupe ne sont donc plus figées dans la logique JavaScript.

### `configureRecognition()`

Configure la reconnaissance vocale avec la langue de réponse déclarée dans `languageCatalog`, ici `fr-FR`.

### `handleUserText()`

Garde la logique de réponse utilisateur et de relance pédagogique, mais utilise désormais la langue française issue du catalogue pour la synthèse vocale de la relance.

## Reconstruction des données d'affichage

Le script principal ne part plus d'un tableau de participants complet écrit en dur.

Il reconstruit les données nécessaires ainsi :

- le nom vient de `characterCatalog` ;
- la langue vient de `languageCatalog` ;
- le fond vient de `placeCatalog` ;
- le texte vient de `scenarioCatalog` ;
- les observations viennent du scénario actif.

Cette logique rend le prototype plus proche d'une future bibliothèque de rencontres plurilingues.

## Limites restantes

- Le scénario actif est encore choisi par `activeScenarioId`.
- La scène collective est encore optimisée pour trois participants.
- Les classes CSS restent liées à certains ids de personnages.
- Il n'existe pas encore de sélection de scénario dans l'interface.
- Il n'existe pas encore de choix d'avatar utilisateur.
- Il n'existe pas encore de vraie bibliothèque de personnages.
