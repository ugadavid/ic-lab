# IC-Lab Hub V0.5 - Visibilite effective et premiers espaces de partage

Date : 2026-07-02

## Objectif

La V0.5 transforme la visibilite de simple metadonnee en premiere regle fonctionnelle de partage. Le hub reste en stockage JSON local, conserve le port `8790`, les comptes demo, les traces V0.3.1, ainsi que les institutions et ownership V0.4.

## Pourquoi appliquer la visibilite maintenant

La V0.4 a pose les champs `ownerId`, `createdBy`, `updatedBy`, `institutionId`, `visibility` et le registre `activity-ownership.json`. La V0.5 utilise ces champs pour determiner quelles activites referencees sont visibles pour un utilisateur connecte, avant une future mutualisation plus large.

## Regles de visibilite

- `private` : visible par le proprietaire et les admins.
- `course` : visible par le proprietaire, les admins, les enseignants des cours ou l'activite est assignee, et les etudiants inscrits a ces cours.
- `institution` : visible par les utilisateurs rattaches a la meme institution et les admins.
- `shared` : visible par les enseignants connectes, les observateurs et les admins.
- `public` : visible par tous les utilisateurs connectes.

La fonction serveur principale est `canViewActivityOwnership(user, ownershipRecord, context)`.

## Endpoints ajoutes ou adaptes

- `GET /api/me/activities` : retourne les activites visibles pour l'utilisateur connecte, avec `visibilityReason`.
- `GET /api/activity-ownership` : retourne tout pour admin, sinon seulement les records visibles.
- `GET /api/activity-ownership/:id` : refuse les records non visibles.
- `PUT /api/activity-ownership/:id` : admin ou proprietaire uniquement.
- `GET /api/sharing-spaces` : expose les premiers espaces de partage.

## Espaces de partage

Fichier : `server/data/sharing-spaces.json`

Espaces seed :

- `space_repli4c` : projet REPLI4C, visibilite `shared`.
- `space_uga` : institution UGA, visibilite `institution`.

Cette structure est preparatoire : elle donne un support aux futurs espaces de mutualisation, sans encore porter une gestion complete des droits.

## Changements admin

Pages V0.5 creees :

- `admin-0.5.html`
- `admin-0.5.css`
- `admin-0.5.js`

L'admin voit toutes les activites referencees, avec proprietaire, institution, visibilite, provenance et une explication theorique de qui peut voir la ressource. Des filtres simples ont ete ajoutes : visibilite, institution, owner, prototype.

## Changements teacher

Pages V0.5 creees :

- `teacher-0.5.html`
- `teacher-0.5.css`
- `teacher-0.5.js`

Une section `Ressources accessibles` affiche les activites visibles pour l'enseignant via `/api/me/activities` :

- activites dont il est proprietaire ;
- activites de cours ;
- activites de meme institution ;
- activites partagees ;
- activites publiques.

Un enseignant proprietaire peut modifier la visibilite d'un record via `PUT /api/activity-ownership/:id`. Un bouton `Assigner` pre-remplit le formulaire d'assignation.

## Changements student

## Limites restantes

- Le modele reste local JSON et mono-process pour cette passe.
- Les espaces de partage sont preparatoires et ne portent pas encore de listes de membres.
- Les permissions restent minimales et pedagogiques, pas un modele juridique complet.
- L'interface de modification ownership reste volontairement simple.
- Les activites visibles via API peuvent inclure des records publics ; l'interface etudiante garde volontairement l'accent sur les activites assignees.

## Lien avec la future migration MariaDB

La migration vers MariaDB n'est pas faite dans cette passe pour conserver un prototype stable et inspectable en JSON. La V0.5 clarifie d'abord les entites et regles : ownership, institutions, espaces de partage et visibilite effective. Ces structures pourront ensuite devenir des tables dans une base `ic_hub`, possiblement sur l'instance deja utilisee par Dico-IC.
