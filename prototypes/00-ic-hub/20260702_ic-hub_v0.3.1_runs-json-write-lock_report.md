# IC-Lab Hub V0.3.1 - Stabilisation runs.json et serveur statique

Date : 2026-07-02

## Objectif

Stabiliser les traces V0.3 du hub IC-Lab sans changer l'architecture : port `8790`, serveur Node local, stockage JSON local, endpoints V0.3 conserves.

## Cause probable de la corruption

Prototype 06 envoie plusieurs evenements de run tres proches au chargement et pendant la rencontre. En V0.3, deux requetes `POST /api/runs/:runId/events` pouvaient lire la meme version de `runs.json`, modifier chacune leur copie, puis ecrire presque simultanement avec le meme fichier temporaire `runs.json.tmp`.

Le symptome observe, `Unexpected non-whitespace character after JSON`, correspond a un fichier JSON devenu invalide apres ces ecritures concurrentes ou partielles.

## Strategie de verrouillage

Ajout d'une queue de promesses par store JSON via `withJsonWriteLock(name, asyncFn)`.

Les sections critiques suivantes sont maintenant serialisees pour `runs.json` :

- creation de run dans `POST /api/runs/start`;
- ajout d'evenement dans `POST /api/runs/:runId/events`.

Chaque endpoint lit, modifie et ecrit `runs.json` dans la section protegee. Deux evenements proches ne peuvent donc plus modifier le fichier en parallele.

## Ecriture atomique

`writeJson()` continue d'ecrire dans un fichier temporaire puis de renommer le fichier. Le nom temporaire est maintenant unique par ecriture :

- PID du process;
- timestamp;
- suffixe aleatoire.

Cela evite que deux ecritures utilisent le meme `.tmp`.

## Recuperation de runs.json corrompu

Au demarrage, le hub valide `runs.json`.

Si le fichier est illisible :

- le fichier corrompu est sauvegarde sous `runs.corrupt.<timestamp>.json`;
- un store propre est recree avec `version: "0.3.1"` et `runs: []`;
- le serveur continue a demarrer.

## Correction de /favicon.ico

`GET /favicon.ico` retourne maintenant explicitement `204 No Content`.

La route ne passe plus par la resolution statique classique quand aucun favicon n'existe.

## Redirections ajoutees

Les anciennes pages principales redirigent vers les pages V0.3 :

- `/student.html` -> `/student-0.3.html`
- `/teacher.html` -> `/teacher-0.3.html`
- `/hub.html` -> `/hub-0.3.html`
- `/admin.html` -> `/admin-0.3.html`

## Protections HTTP statiques

`serveStatic()` verifie `headersSent` et `writableEnded` avant d'ecrire une reponse. Le handler HTTP principal retourne immediatement apres les routes API, ce qui limite les doubles reponses.

## Limites restantes

- Le verrou est en memoire process : il protege ce serveur Node local, pas plusieurs process ecrivant le meme fichier.
- Les traces restent un stockage prototype en JSON local.
- La recuperation de corruption repart d'un store vide pour `runs.json`; le fichier corrompu est conserve pour inspection manuelle.
